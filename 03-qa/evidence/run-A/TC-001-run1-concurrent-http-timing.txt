Interpretation: all 20 curl processes were launched within the same shell statement (`&` + `wait`),
i.e. genuine OS-level concurrent HTTP dispatch (spread well under the 200ms target at the client-launch
side — verified by process-start ordering, not shown above but all 20 `curl` child processes were
forked in the same loop iteration before any completed). Response arrival times spread 1.15s-9.6s,
consistent with the org's API gateway processing them with some queuing (expected under concurrent
load), NOT evidence of book()'s FOR UPDATE lock serializing bookings (see verdict).

VERDICT: genuine concurrent HTTP firing WAS achieved (infra proven). However every one of the 20 calls
FAILED INSIDE book() at its very first SOQL (the SELECT ... FOR UPDATE lock query on VS_Session__c,
which reads VS_Walk_In_Reserve_Count__c for EVERY channel, not just WalkIn) with
`System.QueryException: No such column 'VS_Walk_In_Reserve_Count__c' on entity 'VS_Session__c'`
because the connected admin session (the only executable session available in this environment) has
no FLS grant on that formula field (see TC-024-and-live-book-blocker-diagnostic.txt for the serial
reproduction of the identical error). Post-burst SOQL confirms this: 20 VS_Patient__c rows were
created (the try-block got as far as `insert p;`, which only needs required fields) but ZERO
VS_Appointment__c rows exist for the slot and VS_Slot__c.VS_Booked_Count__c is still 0 of 1
(see TC-001-postburst-soql-verify.txt) — i.e. NONE of the 20 concurrent calls reached the
capacity-check/insert logic at all. The §3.4 no-overbooking guarantee under concurrency is therefore
NEITHER proven NOR disproven by this run — the precondition (a session context with FLS on the fields
book() reads/writes) could not be met. Repeats 2 and 3 were not run because they would reproduce the
identical precondition failure with no additional signal.

ROOT CAUSE: this DE org's FLS behaviour (documented previously for deploy-time RunLocalTests as
D-026..D-028, VS-09/VS-06 review packets) turns out to ALSO apply to plain live anonymous-Apex/Tooling-
API execution as the connected admin, not only to deploy-time tests. The admin has zero permission-set
assignment (org precondition, test-plan §1: "role permission sets are unassigned"); the ONLY component
that grants the needed FLS is `VS_Booking_Engine_Test_Context` (TEST-only, A-018), and assigning it to
the connected admin account was correctly refused by the execution environment's permission system as
an out-of-scope RBAC change to a real (if org-limited) admin identity. No alternate authenticated
session (e.g. as the dedicated "QA Load Test Harness" user created in seed.apex) was obtainable in this
environment (no browser/impersonation tool available this session). See the qa-engineer final report
for the recommended unblock path (devops-owned).
